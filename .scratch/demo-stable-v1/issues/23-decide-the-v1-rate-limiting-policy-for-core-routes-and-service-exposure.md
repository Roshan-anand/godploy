# Decide the V1 rate-limiting policy for core routes and service exposure

Status: hard-human

## What to build

Turn the open V1 rate-limiting note into a concrete product and implementation decision. The result should define which route classes need limits, what the default behavior is for demo-stable installs, and how user-configurable service limits fit into the V1 backlog.

## Acceptance criteria

- [ ] The V1 route classes that need rate limiting are explicitly identified.
- [ ] The default rate-limiting behavior for the demo-stable release is decided and recorded.
- [ ] Any remaining follow-up work for user-configurable service limits is captured as explicit backlog rather than being left implicit.
